It turns out, going through the Twitter Clone code I had written so far and incorporating partials only took a few minutes.  Its pretty easy when there are only a few pages of code to clean up.  Since this was going to be one of the main things I wanted to get done today, I instead dove straight into developing user authorization and authentication.

One of the first decisions I had to address was whether to use a security gem or create my own form of authentication system.  I was leaning toward using the Devise gem but finally decided against it.  The entire reason I am writing this program is to learn how to do things myself.  Why take shortcuts and miss learning important basic features, features that Devise utilizes and depends upon?  While it is a lot more difficult to program my own authentication system, I decided to go for it after all since this is a learning experience.

I came across this wonderful tutorial by Michael Hartl which does an excellent job explaining the intricacies of authentication systems.  I completed Chapter 6 and 7 which runs through setting up a user model, validating 'User' instances, creating a sign-in page and linking this page to the user model.  Bonus, the tutorial uses Bootstrap so it fits in perfectly with what I have already written.

Security for storing passwords was one lesson I was glad to receive.  Security has always been a form of special magic I thought I wouldn't understand for another month or so at least.  After working through some problems though, it seems fairly straightforward.  Rails has a gem called bcrypt which uses an irreversible hash function to convert passwords to encrypted hash values.  This hash value is stored in the database under the appropriate user when they make an account.  When the user attempts to log-in, the password they input is converted to its corresponding hash value and matched to the hash value stored in the database.  The user is allowed access if it matches.  If a hacker were ever to gain access to the database, they would only come away with the stored hash values.  The actual passwords are only used client-side by the users themselves..

When attempting to validate confirmation of a password I was given a brief lesson in regular expressions.  I haven't had too much experience with regular expressions yet but this was a welcome opportunity to expand my boundaries.

/\A[\w+\-.]+@[a-z\d\-.]+\.[a-z]+\z/i

The regular expression above validates the format of an email address.  A breakdown of this regular expression is best summarized at this link.  In the future if ever I am going to work with or practice constructing regular expressions, there is a site called Rubular which is a regular expression editor.  You insert a regular expression and can then test it against a selection of inputted strings.
